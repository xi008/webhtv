# 开发分支与官方同步流程

本文档用于固定 WebHTV 后续二次开发、官方代码同步、targetSdk 适配和发布维护的分支流程。目标是让主线稳定、同步官方时可控、不同 Android targetSdk 路线互不污染。

## 当前仓库定位

正式开发仓库：

```bash
/Users/macbookpro/Desktop/github/webhtv
```

历史基线仓库：

```bash
/Users/macbookpro/Desktop/github/TV
```

`TV` 目录是早期 5.4.1 二开工作区，后续只作为历史参考，不再作为正式功能开发入口。

## 分支职责

### main

`main` 是稳定主线，也是默认开发和默认发布分支。

当前 `main` 已经合并到 `webhtv-latest-target28`，定位是：

```text
官方较新代码 + WebHTV 二开能力 + targetSdk 28
```

后续新增功能默认从 `main` 创建功能分支，测试通过后合回 `main`。

### webhtv-5.4.9

`webhtv-5.4.9` 是 5.4.9 兼容维护分支。

只在以下情况修改：

```text
1. 需要继续发布 5.4.9 兼容版
2. 需要修复严重问题
3. 需要把非常稳定的小功能补到旧版
```

不要把日常新功能优先开发在这个分支。

### webhtv-latest-target28

`webhtv-latest-target28` 是最新官方代码在 targetSdk 28 路线上的参考分支。

当前 `main` 已经等同于它。后续如果 `main` 继续作为 targetSdk 28 稳定主线，这个分支可以作为阶段性同步点或备份点，不作为日常功能开发入口。

### webhtv-latest-target37

`webhtv-latest-target37` 是新版 targetSdk 适配分支。

它用于验证官方新 targetSdk 带来的权限、存储、后台、WebView、安装兼容等变化。只有充分测试后，才考虑把 `main` 切换到这条路线。

### upstream/fongmi

官方原版仓库只作为代码来源，不在官方分支上做 WebHTV 功能开发。

推荐远端命名：

```bash
git remote add fongmi https://github.com/FongMi/TV.git
```

如果已经存在，直接使用现有远端即可。

## 日常新增功能流程

新增功能基于 `main` 开发。

```bash
cd /Users/macbookpro/Desktop/github/webhtv
git fetch origin
git switch main
git pull --ff-only
git switch -c feat/功能名
```

完成开发和验证后：

```bash
git switch main
git merge --ff-only feat/功能名
git push origin main:main
```

如果功能分支中包含多个提交，且希望保留功能开发过程，可以使用普通 merge：

```bash
git switch main
git merge --no-ff feat/功能名
git push origin main:main
```

如果该功能也需要同步到 `webhtv-latest-target37`：

```bash
git switch webhtv-latest-target37
git pull --ff-only
git merge main
git push origin webhtv-latest-target37:webhtv-latest-target37
```

也可以只挑选某几个功能提交：

```bash
git switch webhtv-latest-target37
git cherry-pick <commit>
git push origin webhtv-latest-target37:webhtv-latest-target37
```

规则：功能从 `main` 单向同步到 `webhtv-latest-target37`，不要把 `webhtv-latest-target37` 反向合回 `main`，除非明确决定主线升级到新版 targetSdk。

## 官方代码同步流程

同步官方代码不要直接在 `main` 上硬合，先创建临时集成分支。

### targetSdk 28 稳定路线

```bash
cd /Users/macbookpro/Desktop/github/webhtv
git fetch origin
git fetch fongmi
git switch main
git pull --ff-only
git switch -c sync/fongmi-版本号-target28
```

然后把官方更新合入这个临时分支。

如果是按官方提交链逐步合并，使用：

```bash
git cherry-pick <官方提交1>
git cherry-pick <官方提交2>
git cherry-pick <官方提交3>
```

如果确认可以整体合并，使用：

```bash
git merge <官方版本提交或标签>
```

合并时必须保持以下原则：

```text
1. 保留 WebHTV 已实现的二开功能
2. 保留 targetSdk 28 路线需要的构建配置
3. 保留 clone 后可直接编译所需的 app/libs 和 third_party/maven
4. 不恢复官方自动更新弹窗
5. 不覆盖 WebHome、pan、check、debug、增强设置等开放能力
6. 不把新版 targetSdk 强行带入 main，除非明确决定主线升级
```

集成完成后先验证构建：

```bash
bash gradlew :app:assembleMobileArm64_v8aRelease
bash gradlew :app:assembleLeanbackArm64_v8aRelease
bash gradlew :app:assembleLeanbackArmeabi_v7aRelease
```

验证通过后再合回 `main`：

```bash
git switch main
git merge --no-ff sync/fongmi-版本号-target28
git push origin main:main
```

### targetSdk 37 适配路线

新版 targetSdk 单独在 `webhtv-latest-target37` 或临时分支上处理。

```bash
cd /Users/macbookpro/Desktop/github/webhtv
git fetch origin
git fetch fongmi
git switch webhtv-latest-target37
git pull --ff-only
git switch -c sync/fongmi-版本号-target37
```

合入官方更新后，重点验证：

```text
1. 手机端安装和启动
2. 电视端安装和启动
3. WebHome 页面加载、透明背景、返回、恢复显示
4. pan.play、check.links、debug 日志
5. 爬虫、Jar、站点配置、WebView Cookie
6. 文件访问、下载、缓存、外部播放器
7. Android 新权限、后台服务、通知、存储行为
```

验证通过后合回 `webhtv-latest-target37`：

```bash
git switch webhtv-latest-target37
git merge --no-ff sync/fongmi-版本号-target37
git push origin webhtv-latest-target37:webhtv-latest-target37
```

不要因为 `webhtv-latest-target37` 通过构建就立即合回 `main`。它需要经过实际设备测试和功能回归。

## main 何时切到新版 targetSdk

只有满足以下条件，才考虑让 `main` 从 targetSdk 28 切到新版 targetSdk：

```text
1. 手机端和电视端都能稳定安装、启动、播放
2. WebHome、透明背景、Nostr 主页、盘搜、最近观看都正常
3. pan.play 和 push 播放链路都正常
4. 网盘检测、debug 日志、增强设置正常
5. 官方新代码功能没有明显回退
6. Android 新权限导致的问题已经处理
7. 老设备兼容风险可以接受
```

确认切换时再执行明确的主线升级：

```bash
git switch main
git merge --no-ff webhtv-latest-target37
git push origin main:main
```

这属于版本路线决策，不是日常开发流程。

## 版本发布流程

发布前确认当前分支：

```bash
git branch --show-current
```

确认版本号：

```bash
rg -n "versionCode|versionName|targetSdk" app/build.gradle gradle/libs.versions.toml
```

常用构建命令：

```bash
bash gradlew :app:assembleMobileArm64_v8aRelease
bash gradlew :app:assembleLeanbackArm64_v8aRelease
bash gradlew :app:assembleLeanbackArmeabi_v7aRelease
```

发布前建议至少验证：

```text
1. 手机端 release 包可安装
2. 电视端 64 位 release 包可安装
3. 电视端 32 位 release 包可安装
4. App 首页、搜索、播放、设置页正常
5. WebHome 自定义主页正常
6. clone 后不依赖本机 local.properties 也能构建
```

打 tag 示例：

```bash
git tag v版本号-日期时间
git push origin v版本号-日期时间
```

## 文档维护规则

涉及开放能力、WebHome SDK、pan、check、debug、设置项、构建流程的变更，需要同步更新文档。

原则：

```text
1. 文档写最终状态，不写“以前怎样、现在怎样”的流水账
2. 开发者可以复制命令直接执行
3. 不依赖根目录重复文档
4. 以 docs/应用完整开发文档.md 作为能力说明主文档
5. 本文档只维护分支、同步、发布流程
```
