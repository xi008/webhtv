# 播放记录开放与上报方案

状态：已实现  
日期：2026-06-12

## 1. 目标与入口

播放记录开放只提供两种入口：

1. **App 配置 Webhook**：用户在增强功能里本地配置端点，App 在播放过程中主动 POST 到用户服务器。
2. **爬虫调用 API**：Spider/爬虫在自身代码里调用受限 API 读取当前播放记录，再自行处理或上报。

不提供第三种入口。远程点播配置、站点配置、CSP、`extensions`、`webHomeExtensions` 都不能静默下发播放记录 Webhook；WebHome 既有 `fm.history()`/`app.history` 属于原有历史能力，不是本方案新增入口。

## 2. 当前实现位置

核心实现位于 `app/src/main/java/com/fongmi/android/tv/playback/`：

| 组件 | 职责 |
| --- | --- |
| `PlaybackRecord` | 从 `History + PlayerManager` 生成播放记录 DTO，并生成 `eventId/sessionId/dedupeKey/clientKey` |
| `PlaybackFieldPolicy` | 控制 API 和 Webhook 字段输出范围 |
| `PlaybackApi` | App 内部只读 helper，限制当前调用方站点和当前播放项 |
| `PlaybackEventCollector` | 从播放状态和进度保存点收集播放事件 |
| `PlaybackWebhookSender` | 异步发送、节流、重试、串行化和请求头封装 |
| `PlaybackWebhookStore` / `WebhookConfig` | 本地 Webhook 配置存储、失败暂停和旧配置迁移 |
| `server.process.PlaybackRecordApi` | 面向 JS/Python/PHP/Jar 等爬虫的本机 HTTP REST API |

配置界面为 `PlaybackWebhookDialog`，入口在增强功能页“播放记录上报”。

## 3. 爬虫 HTTP API 文档

爬虫侧使用本机 HTTP REST API 获取当前播放记录，适用于 JS、Python、PHP、Jar 等不同运行时。Java `PlaybackApi.current(siteKey)` 仅作为 App 内部复用 helper，不作为对外接入方式。

API 只读当前播放项，不提供全局历史列表。调用时必须传 `siteKey`，App 会校验该 `siteKey` 与当前播放记录一致，避免一个站点读取另一个站点的当前播放数据。

### 3.1 Endpoint

```http
GET http://127.0.0.1:{port}/api/playback/current?siteKey={siteKey}
```

兼容短路径：

```http
GET http://127.0.0.1:{port}/playback/current?siteKey={siteKey}
```

说明：

- `{port}` 为 App 本机 HTTP 服务端口。JS Spider 可用 `getPort()` 获取；Jar 爬虫可用 `com.github.catvod.Proxy.getPort()` 获取；其它运行时只要能访问 App 本机或局域网 HTTP 地址即可。
- 也可以使用局域网地址：`http://设备IP:{port}/api/playback/current?siteKey={siteKey}`。
- 接口支持 `GET` 和 `POST`，推荐用 `GET`。
- 返回内容是播放记录 JSON 本体，不包 `code/data` 外壳，方便不同语言直接解析。

### 3.2 参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `siteKey` | string | 是 | 调用方站点 key。必须与当前播放记录的 `siteKey` 完全一致 |
| `site` | string | 否 | `siteKey` 的兼容别名；同时存在时优先使用 `siteKey` |

### 3.3 HTTP 状态码

| 状态码 | 含义 | Body |
| --- | --- | --- |
| `200` | 成功读取当前播放记录 | 播放记录 JSON 本体 |
| `400` | 缺少 `siteKey` | `{"code":400,"message":"siteKey不能为空"}` |
| `404` | 当前无可读播放记录，或当前播放项不属于该站点 | `{"code":404,"message":"当前无可读播放记录"}` |
| `405` | 方法不支持 | `{"code":405,"message":"只支持 GET/POST"}` |
| `500` | 内部错误 | `{"code":500,"message":"..."}` |

### 3.6 返回空的情况

- `siteKey` 为空。
- 当前是隐身模式。
- 当前没有有效播放记录。
- 当前播放项不属于调用方站点。
- 当前播放记录还没有可保存的 `positionMs` / `durationMs`。

### 3.7 API 字段范围

爬虫 HTTP API 返回 `apiSafe` 字段集：`schema/timestamp/sessionId/dedupeKey/cid/historyKey/siteKey/siteName/vodId/vodName/vodPic/flag/episodeName/state/positionMs/durationMs/progress/speed/completed`。

API 不返回 `event`、`eventId`、`episodeUrl`、真实播放地址、headers、cookies、配置地址、Jar/ext、设备标识、全局历史或其他站点历史。

## 4. App Webhook 配置

Webhook 是用户本地配置，配置项包括：

| 配置项 | 默认值 | 说明 |
| --- | --- | --- |
| `name` | 空 | 可选；为空时 UI 使用端点 URL 的 host 作为名称，不包含协议和 path |
| `enabled` | true | 启用/停用端点 |
| `url` | 空 | 必填，必须以 `http` 开头 |
| `token` | 空 | 可选；仅作为 `X-WebHTV-Token` 请求头发送，不参与 `dedupeKey` 计算 |
| `events` | `progress, ended` | 上报时机，UI 中以“基础/标准/完整”三档呈现 |
| `siteKeys` | 空 | 可选；为空表示全部站点 |
| `fieldPreset` | `safe` | 字段预设 |
| `fields` | 空 | 仅自定义预设使用，由 UI 多选，不使用手填 key |
| `progressIntervalSec` | 30 | progress 节流间隔，0 表示关闭周期进度事件 |
| `maxRetries` | 2 | 失败重试次数，范围 0-3 |

连续失败达到阈值后端点会自动暂停。用户再次点击“启用”或保存启用状态时，会清空暂停和最近错误。

## 5. 上报时机

UI 不直接暴露所有底层事件，而是使用三档中性配置：

| 档位 | 实际事件 | 适用场景 |
| --- | --- | --- |
| 基础 | `progress`, `ended` | 默认推荐，适合播放记录同步。暂停或退出时仍会补发一次最终 `progress` |
| 标准 | `start`, `progress`, `ended` | 比基础多播放开始事件，同时避免暂停/恢复噪声 |
| 完整 | `start`, `progress`, `stop`, `ended` | 需要区分主动退出/停止和自然完播 |

底层仍保留 `pause/resume` 事件常量用于兼容或后续扩展，但默认 UI 不再暴露它们。原因是对播放记录同步而言，暂停/恢复通常只会增加噪声；关键状态可以通过最终 `progress`、`stop`、`ended` 表达。

## 6. 字段预设与字段说明

| 预设 | 字段 | 说明 |
| --- | --- | --- |
| `safe` | 协议字段 + 播放对象基础信息 + 播放进度 | 默认推荐 |
| `standard` | `safe` + `appVersion` + `client` + `clientKey` | 便于服务端排查不同客户端 |
| `anonymous` | 协议字段 + `historyKey` 的 SHA256 + 播放进度 | 不包含片名、封面、站点名等可识别字段 |
| `custom` | 协议字段固定携带，其他字段由用户多选 | UI 只展示可选 key，不要求用户手填 |

协议字段包括：`schema/event/eventId/timestamp/sessionId/dedupeKey`。

当前实现不开放真实播放地址、headers、cookies、配置地址、Jar/ext、本地路径、IP/MAC 等 L4 高风险字段。

### 6.1 字段字典

| 字段 | 类型 | 出现条件 | 说明 |
| --- | --- | --- | --- |
| `schema` | string | API、所有 Webhook 预设 | 固定为 `webhtv.playback.v1`，用于服务端识别协议版本 |
| `event` | string | Webhook | 事件名，格式为 `playback.<type>`；爬虫 API 不返回 |
| `eventId` | string | Webhook | 单次事件 UUID；同一次事件重试保持不变，可用于幂等去重 |
| `timestamp` | number | API、所有 Webhook 预设 | App 生成记录的毫秒时间戳，Unix epoch milliseconds |
| `sessionId` | string | API、所有 Webhook 预设 | 本次播放会话 UUID；同一播放条目连续播放期间保持稳定，切换条目后更新 |
| `dedupeKey` | string | API、所有 Webhook 预设 | 播放条目稳定去重 key，SHA256；由播放对象字段生成，不包含 token |
| `cid` | number | `safe`、`standard`、`custom`、API | 当前点播配置分类 id，用于区分本地配置空间 |
| `historyKey` | string | `safe`、`standard`、`custom`、API；`anonymous` 中为 SHA256 | App 历史记录主键，通常由 `siteKey@@@vodId` 组成；匿名预设只发哈希 |
| `siteKey` | string | `safe`、`standard`、`custom`、API | 站点 key，来自当前点播站点配置 |
| `siteName` | string | `safe`、`standard`、`custom`、API | 站点显示名称；取不到名称时回退为 `siteKey` |
| `vodId` | string | `safe`、`standard`、`custom`、API | 影片 id，从 `historyKey` 第二段解析 |
| `vodName` | string | `safe`、`standard`、`custom`、API | 影片名称 |
| `vodPic` | string | `safe`、`standard`、`custom`、API | 影片封面 URL，可能为空 |
| `flag` | string | `safe`、`standard`、`custom`、API | 当前播放线路/来源名称 |
| `episodeName` | string | `safe`、`standard`、`custom`、API | 当前集名称，对应历史记录 `vodRemarks` |
| `episodeUrl` | string | `custom` 可选 | 当前集原始 URL。默认 `safe` / `standard` 不发送；仅用户在自定义字段中明确选择时发送 |
| `episodeIndex` | number | `custom` 可选 | 当前集序号。当前实现预留字段，无法识别时不输出 |
| `state` | string | API、所有 Webhook 预设 | 播放器状态：`idle`、`buffering`、`playing`、`paused`、`ended` |
| `positionMs` | number | API、所有 Webhook 预设 | 当前播放进度，毫秒 |
| `durationMs` | number | API、所有 Webhook 预设 | 当前视频总时长，毫秒 |
| `progress` | number | API、所有 Webhook 预设 | `positionMs / durationMs`，范围 `0` 到 `1`，保留 4 位小数 |
| `speed` | number | API、所有 Webhook 预设 | 当前播放倍速 |
| `completed` | boolean | API、所有 Webhook 预设 | 是否可视为完播；播放器 ended，或进度达到 95% 以上时为 true |
| `appVersion` | string | `standard`、`custom` 可选 | App 版本名，便于服务端排查客户端版本差异 |
| `client` | string | `standard`、`custom` 可选 | 构建 flavor，例如 mobile/leanback 对应的客户端类型 |
| `clientKey` | string | `standard`、`custom` 可选 | 本机 Android ID 的 SHA256。默认 `safe` 不发送，避免默认携带设备级标识 |

### 6.2 预设字段集合

| 预设 | 字段集合 |
| --- | --- |
| `safe` | `schema/event/eventId/timestamp/sessionId/dedupeKey/cid/historyKey/siteKey/siteName/vodId/vodName/vodPic/flag/episodeName/state/positionMs/durationMs/progress/speed/completed` |
| `standard` | `safe` + `appVersion/client/clientKey` |
| `anonymous` | `schema/event/eventId/timestamp/sessionId/dedupeKey/historyKey(sha256)/state/positionMs/durationMs/progress/speed/completed` |
| `custom` | 协议字段固定发送：`schema/event/eventId/timestamp/sessionId/dedupeKey`；其余字段由 UI 多选，范围见字段字典 |

## 7. Webhook 接收 API 文档

Webhook 是 App 主动调用用户配置的 HTTP 接口。用户服务器只需要实现一个能接收 JSON 的 `POST` 端点。

### 7.1 Endpoint

```http
POST <用户在 App 中填写的端点 URL>
Content-Type: application/json
```

要求：

- URL 必须以 `http://` 或 `https://` 开头。
- App 不要求固定 path，用户可自行使用 `/api/webhtv/playback`、`/webhook/playback` 等路径。
- Token 可选；如果填写，App 只通过 `X-WebHTV-Token` 请求头发送，不参与 payload 计算。
- App 认为任意 `2xx` HTTP 状态码为成功；非 `2xx`、网络错误或超时都按失败处理并进入重试。

### 7.2 请求头

```http
Content-Type: application/json
X-WebHTV-Timestamp: 1781170000
X-WebHTV-Token: <用户可选 token>
X-WebHTV-Webhook-Id: <eventId>
X-WebHTV-Dedupe-Key: <dedupeKey>
Idempotency-Key: <eventId>
```

| 请求头 | 必填 | 说明 |
| --- | --- | --- |
| `Content-Type` | 是 | 固定为 `application/json` |
| `X-WebHTV-Timestamp` | 是 | 发送请求时的秒级 Unix 时间戳 |
| `X-WebHTV-Token` | 否 | 用户在 App 配置的 token。服务端可用它做用户空间、鉴权或分组 |
| `X-WebHTV-Webhook-Id` | 是 | 与 payload 的 `eventId` 一致 |
| `X-WebHTV-Dedupe-Key` | 是 | 与 payload 的 `dedupeKey` 一致 |
| `Idempotency-Key` | 是 | 与 `eventId` 一致，用于兼容常见 Webhook 幂等处理 |

`token` 只是用户服务器分组/鉴权用的共享值，App 不用它计算任何字段。同一个 token 下，服务端可以用 `dedupeKey` 合并多端同一播放条目，用 `eventId`/`Idempotency-Key` 去重同一次事件重试。

### 7.3 事件名

| 事件 | UI 档位 | 触发说明 |
| --- | --- | --- |
| `playback.start` | 标准、完整 | 播放条目首次进入播放状态时发送 |
| `playback.progress` | 基础、标准、完整 | 周期进度事件；默认 30 秒节流。暂停、退出、完播前会补发最终进度 |
| `playback.stop` | 完整 | 主动退出/停止播放时发送 |
| `playback.ended` | 基础、标准、完整 | 播放器自然完播时发送 |

底层保留 `playback.pause` / `playback.resume`，但当前 UI 不暴露，默认不建议服务端依赖这两个事件。

### 7.5 推荐服务端处理

服务端建议按下面顺序处理：

1. 校验 `Content-Type` 和 JSON body。
2. 如果服务端要求鉴权，校验 `X-WebHTV-Token` 是否属于当前用户。
3. 用 `eventId` 或 `Idempotency-Key` 做事件级幂等，重复事件直接返回 `2xx`。
4. 用 `dedupeKey` 在用户空间内合并同一播放条目。
5. 按 `event` 更新状态：`progress` 更新进度，`ended` 标记完播，`stop` 标记主动退出。

最小 Node.js/Express 示例：

```js
app.post('/api/webhtv/playback', express.json(), async (req, res) => {
  const token = req.header('X-WebHTV-Token') || '';
  const idempotencyKey = req.header('Idempotency-Key') || req.body.eventId;
  const dedupeKey = req.body.dedupeKey;

  if (!dedupeKey || !idempotencyKey) return res.sendStatus(400);
  if (await seenEvent(idempotencyKey)) return res.sendStatus(204);

  await markEventSeen(idempotencyKey);
  await upsertPlayback({ token, dedupeKey, payload: req.body });
  res.sendStatus(204);
});
```

## 8. 去重与身份

- `eventId`：每次事件生成一个 UUID，用于投递幂等和重试去重。
- `dedupeKey`：由播放对象稳定字段生成的 SHA256，用于同一播放条目的合并，不包含 token。
- `clientKey`：本机 Android ID 的 SHA256，仅在 `standard/custom` 字段策略中可发送，默认 `safe` 不发送。

服务端推荐策略：

1. 先按 `X-WebHTV-Token` 或 endpoint 自身归属确定用户空间。
2. 再按 `dedupeKey` 合并播放条目。
3. 再按 `eventId`/`Idempotency-Key` 去重投递。

## 9. 发送策略

- 发送在后台线程执行，不阻塞播放和历史保存。
- `progress` 默认 30 秒节流。
- 暂停、退出、完播时会取消待发送的旧 progress，并在需要时立即补发最终 progress。
- 单个 endpoint 串行发送，避免乱序；不同 endpoint 可并发。
- HTTP 超时为 5 秒。
- 失败最多重试 0-3 次，指数退避。
- 连续失败达到阈值后暂停端点，避免持续打扰服务器或耗电。

## 10. 隐私声明

首次打开配置时显示隐私声明，说明播放记录 Webhook 会把影片名称、进度等观看记录发送到用户配置的服务器。App 不会收集或转发给第三方；用户可随时停用或删除端点。

## 11. 当前结论

- 入口只有 App Webhook 和爬虫 API。
- 不存在 CSP 内联配置播放上报入口。
- Token 可选，只随请求头发送，不参与 dedupe 计算。
- 默认上报时机为“基础”：进度更新 + 完播。
- 默认字段预设为 `safe`。
- 名称可选，默认显示端点域名。
- 自定义字段通过 UI 多选，不需要用户手填 key。
